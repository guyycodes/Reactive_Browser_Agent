================================================================================
 0. MENTAL MODEL (read this first if "agentic code" is new)
================================================================================

An AGENT is just a program that:
  (1) takes a goal ("reset Jane's password");
  (2) uses an LLM to reason about HOW to achieve the goal;
  (3) calls TOOLS (RAG search, browser automation, etc.) to observe the
      world and take actions;
  (4) reports back to a human WHO decides whether to proceed.

This particular agent adds TWO product constraints on top:
  • Every run is AUDITABLE — every thought, tool call, and observation is
    emitted as an event frame, streamed to a reviewer UI, AND persisted to
    Postgres. Nothing is lost between runs.
  • Every destructive action passes through a HUMAN GATE — a step that
    blocks the workflow until a reviewer approves, rejects, or asks the
    agent to re-plan. Two gates exist: pre-execution ("does this plan look
    right?") and post-execution ("did it actually work?").

                                  ┌─────────┐
  Raw ticket "reset Jane's       │         │
  password" ────── POST /triage ─▶│  Agent  │── WebSocket /stream/:id ──▶ UI
                                  │         │     (envelope frames)
  Human decision ───── POST    ──▶│         │── HTTP to RAG / LLM / Browser ─▶
  /runs/:id/review                 └─────────┘

This file explains what's INSIDE that box.

Rules of thumb for beginners:
  • Never import "up the stack." A tool wrapper should not import from a
    workflow. A primitive should not know about the workflow engine.
  • Every cross-layer call emits ≥1 envelope frame so the UI can see it
    happen. This is the "frames are the source of truth" invariant.
  • Every layer has a Zod schema at its boundary so bad data fails loud
    instead of silently.


================================================================================
 2. CORE CONCEPTS (one paragraph each, for beginners)
================================================================================

2.1  WORKFLOW STEP
    A named unit of work with typed input/output. The agent has 9 of them
    (see §3). Each step is a function registered with Mastra via
    `createStep({ id, inputSchema, outputSchema, execute })`. Each step
    automatically emits `step.started` / `step.completed` / `step.failed`
    frames. Output of step N becomes input of step N+1.

2.2  ENVELOPE FRAME
    A JSON object with a `type` discriminator (e.g. `llm.thinking.delta`,
    `tool.completed`, `review.requested`) and a small set of header fields
    (runId, seq, stepId, ts). Every interesting event inside the agent
    becomes a frame. Frames are capped at 16 KiB, validated by Zod,
    persisted to Postgres, and streamed to the UI via WebSocket. The full
    list lives in `src/events/envelope.ts`.

2.3  EVENTBUS
    A per-run in-memory ring buffer that:
      • assigns monotonic `seq` numbers so the UI never sees gaps;
      • fans frames out to any subscribed WebSocket clients;
      • brokers human review decisions from the UI back into the workflow
        (gate suspend / resume via per-stepId decision slots).
    Lives in `src/events/bus.ts`.

2.4  RUNCONTEXT
    Per-run ambient data (runId, bus, ticket, browser session, prior
    observations) carried through `await` boundaries via Node's
    AsyncLocalStorage. Think of it as "thread-local for async code." Any
    step or tool wrapper can call `getRunContext()` and see the same
    object. Lives in `src/mastra/runContext.ts`.

    CTX SPREAD INVARIANT (critical — read the docblock in that file):
    `withRunContext({ ...ctx, field: v }, fn)` creates a NEW object.
    Mutations inside the spread are LOST when the scope unwinds. Only
    certain fields are safe to spread around (the file enumerates them).
    Violations caused 3 of the 4 bugs we hunted during the 7b.iii.b
    series.

2.5  REACT PATTERN (THE big idea)
    "Reasoning + Acting." Instead of the LLM emitting an answer in one
    shot, it runs a LOOP:
        think → call a tool → observe the result → think again
    ...until it has enough information to produce a final answer. One
    reusable factory (`createReActStep` in `src/mastra/lib/reactRunner.ts`)
    implements this loop. You apply it to a step by listing the tools
    that step can use. See §5 for the visual.

2.6  GATE (HUMAN SUSPENSION POINT)
    A workflow step whose `execute` function calls
    `bus.awaitDecisionForStep(runId, stepId)` — which blocks until a human
    posts a decision via HTTP or WebSocket. Two gates exist: pre-exec
    (`review_gate`) and post-exec (`human_verify_gate`). Gates are
    where the product's "human-in-the-loop" guarantee lives. See §6 for
    the two meta-loops that gates can trigger.

2.7  CIRCUIT BREAKER
    A named reliability wrapper around unreliable external calls (mainly
    the Anthropic API). Three-state machine: closed (normal) → open
    (failing, short-circuit to error) → half-open (probe). Automatic
    exponential-backoff retry on transient 5xx. Lives in
    `src/lib/circuit.ts`. Get one by name: `getCircuit("anthropic")`.

2.8  META-LOOP
    A loop at the WORKFLOW level (not the step level) that re-runs
    multiple steps with a reviewer's feedback incorporated. Two exist:
    pre-exec refine (reviewer clicks Edit with a note → re-run Block 1)
    and post-exec backtrack (reviewer rejects post-exec verification →
    re-run the full pre-notify chain). See §6.

================================================================================
 4. BLOCK 1 — THE COGNITIVE SUB-WORKFLOW
================================================================================

Block 1 is the first workflow step BUT it's actually a little iteration
controller internally. It runs classify → retrieve → plan → dry_run up
to 3 times, carrying observations forward if the first pass didn't
produce a viable plan.

Why: sometimes Sonnet says "I need more info" on its first planning pass.
Instead of giving up, we let it see its own failure reason and try again
with that context.

Block 1 is NOT a Mastra sub-workflow — it's a plain-TypeScript orchestrator
(`src/mastra/lib/blockController.ts`) that invokes the extracted step
bodies directly. It emits its OWN step.started/completed frames under
their original stepIds (classify, retrieve, etc.) so the reviewer UI
sees each sub-step as usual.

                     block1Step.execute
                          │
                          ▼
                 runBlock1(input, deps, { seedObservations? })
                          │
                          │ for each pass 0..BLOCK1_MAX_PASSES-1:
                          │
                          ├─▶ emit block.iteration.started
                          │
                          │  ┌──────────────────────────────────────────┐
                          │  │ withRunContext({ ...ctx,                 │
                          │  │   priorObservations: [...observations]}) │
                          │  │                                          │
                          │  │   runClassifyStep   (Haiku)              │
                          │  │      ↓                                   │
                          │  │   runRetrieveStep   (ReAct, RAG)         │
                          │  │      ↓                                   │
                          │  │   runPlanStep       (Sonnet + thinking)  │
                          │  └──────────────────────────────────────────┘
                          │    ↑ cognitive spread — safe because these
                          │      three only READ from ctx
                          │
                          │  Exit check:
                          │    plan.requiresContext?  → record "plan_requires_context"
                          │    plan.actions==[]?      → record "plan_empty_actions"
                          │    else run dry_run:
                          │         ↓
                          │    runDryRunStep    (Playwright, OUTER ctx scope)
                          │                      ↑
                          │                      │
                          │                      └── DO NOT spread here — dry_run
                          │                          MUTATES ctx.browser. See
                          │                          CTX SPREAD INVARIANT in
                          │                          runContext.ts.
                          │
                          │    dry_run.domMatches?  →  exit_signal_ok (break loop)
                          │    else                →  record "dry_run_mismatch"
                          │
                          ├─▶ emit block.iteration.completed (with reason)
                          │
                          └─▶ next pass, with accumulated observations
                              prepended to the next pass's prompts

        On exhaust (3 passes, no exit): return Block1Result with
        `passedLast: false` + `allReasons: [...]`. DryRunSchema.blockResult
        is set; reviewer UI shows an "exhausted" banner and disables the
        Approve button.

Design invariants (both documented in `blockController.ts` docblock):
  #1  stepId namespace — inner steps emit under their ORIGINAL stepIds
      via the ambient bus. "block1" is only used for the wrapping step.*
      frames + the block.iteration.* frames.
  #2  context-mutation scope — dry_run runs OUTSIDE the cognitive spread.
      If you add a new inner step that MUTATES ctx, do NOT place it
      inside a spread. See CTX SPREAD INVARIANT.

 5. THE REACT PATTERN (the big one)
================================================================================

This is the reusable primitive that lets an LLM step call tools in a loop.
Right now it's applied to ONLY the `retrieve` step. Over time you'll apply
it to classify, plan, and verify too (per MASTER_PLAN §4.1).

             │                                          │
             │     ┌──────────────────────────────┐     │

             │     └──────────────────────────────┘     │
             │                │                         │
             │                ▼                         │
             │     ┌──────────────────────┐             │
             │     │  streamMessage(      │             │
             │     │    system,           │             │
             │     │    messages,         │ ← builds user msg
             │     │    tools,            │   from buildUserMessage
             │     │    thinkingEnabled   │   callback (which reads
             │     │  )                   │   ambient priorObservations)
             │     └──────────────────────┘             │
             │                │                         │
             │                │ emits llm.thinking.delta│
             │                │ emits llm.text.delta    │
             │                │ emits llm.tool_use.*    │

             │    ┌───────────────────────────┐         │

             │       │                  to messages     │

Anatomy of ONE iteration's frame stream (reviewer UI sees this):
      react.iteration.started { stepId: "retrieve", iteration: 0 }
         llm.message.started   { model: "sonnet", thinkingEnabled: true }
         llm.thinking.delta    "I should search runbooks for..."
         llm.thinking.delta    "...password reset procedures..."
         llm.text.delta        "Let me call rag.retrieveRunbooks."
         llm.tool_use.started  { name: "rag_retrieveRunbooks" }
         llm.tool_use.completed
         llm.message.completed { stopReason: "tool_use" }
       tool.started            { name: "rag.retrieveRunbooks" }
       rag.retrieved           { hits: [3 items] }
       tool.completed          { durationMs: 116 }
      react.iteration.completed { final: false,
                                  toolUsed: "rag_retrieveRunbooks",
                                  observationSummary: "3 hits, top score 0.76" }

Every frame inside an iteration carries an optional `reactIterationId`
so the reviewer UI can nest them under a single collapsible row.

Where it lives:
  The runner                     → src/mastra/lib/reactRunner.ts
  First (and currently only) use → src/mastra/workflows/triage.ts:
                                    runRetrieveStep (look for
                                    `createReActStep({ id: "retrieve" })`)

How to ADD a ReAct step (checklist):
  1. In `triage.ts`, define `runMyStep` using `createReActStep({...})`.
  2. Pass it a list of tools from `src/mastra/tools/`.
  3. Provide `buildSystem` and `buildUserMessage` callbacks — these read
     from `getRunContext()` to include ticket + prior observations.
  4. Set `maxIterations` (typically 3-5).
  5. Register the resulting step in the Mastra workflow chain in the same
     file (or wrap it in blockController if it's part of Block 1).
  6. No envelope changes needed — the runner's frames are already wired.

 6. THE TWO META-LOOPS (human-guided re-planning)
================================================================================

Beyond the single-step ReAct loop above, the WORKFLOW itself can loop at
two places when a human asks for it. These are the agent's "conversational"
surfaces — the product's pitch lives here.


6.1  PRE-EXEC REFINE (triggered by reviewer clicking EDIT at review_gate)
─────────────────────────────────────────────────────────────────────────

                 reviewGateStep (emits review.requested, awaits)
                           │
                           ▼
               ┌─────────────────────────┐
               │ Reviewer sees the plan  │
               │ Clicks EDIT + types     │
               │ "use the Unlock runbook"│
               └─────────────────────────┘
                           │
                           ▼  WS review.decide{decision:"edit",patch.notes:"..."}
                  bus commits decision
                           │
                           ▼
            reviewGateStep's refine loop kicks in:
                           │
                           ├─▶ buildPreGateRefineContext(notes, N)
                           │      → ["Pre-exec refine 1: reviewer note: ..."]
                           │
                           ├─▶ emit block.backtrack.triggered
                           │      { fromStep:"review_gate", backtrackCount:1 }
                           │
                           ├─▶ emit synthetic block1 step.started
                           │      (so LEFT column updates)
                           │
                           ├─▶ runBlock1(ticket, deps, {
                           │         seedObservations: carriedContext  ← NOTE
                           │      })                                   ← NOTE
                           │
                           │   NOTE: "seedObservations" is how you thread
                           │   observations INTO Block 1. DO NOT wrap
                           │   runBlock1 in a withRunContext spread.
                           │   That pattern was doubly broken and is
                           │   banned — see CTX SPREAD INVARIANT.
                           │
                           ├─▶ emit synthetic block1 step.completed
                           │      (so LEFT column sees the REFINED plan)
                           │
                           ├─▶ emit fresh review.requested
                           │
                           └─▶ await next decision (4-decision model — Week-2a):
                                 ├─ approve   → proceed to executeStep
                                 ├─ edit      → refine loop with reviewer's
                                 │              patch.notes threaded as seed
                                 │              observations
                                 ├─ reject    → refine loop with auto-generated
                                 │              directive seed ("take a
                                 │              different approach")
                                 │              NOTE: reject was pre-week2a
                                 │              a terminate; week2a widened
                                 │              the enum so reject now means
                                 │              "replan without notes" and
                                 │              terminate is the explicit
                                 │              full-stop below
                                 ├─ terminate → skip cascade to
                                 │              logAndNotifyStep
                                 │              (status=rejected)
                                 └─ cap trip  → returns decision=terminate
                                              (MAX_PRE_GATE_REFINES=2 shared
                                               budget across edit+reject)

                           ├─▶ buildBacktrackContext(currentVerify, note, N)
                           │      → ["Backtrack 1: the human rejected the
                           │          post-execution result.",
                           │         "Reviewer note: ...",
                           │         "Prior attempt's execute.stepsRun: 4",
                           │         "Prior attempt's verify.success: true",
                           │         "Prior verify evidence: ..."]
                           │
                           ├─▶ emit block.backtrack.triggered
                           │      { fromStep:"human_verify_gate",
                           │        backtrackCount:1 }
                           │
                           ├─▶ synthetic block1 step.started / completed
                           │      (so LEFT column updates on each backtrack)
                           │
                           ├─▶ runBlock1(..., { seedObservations })
                           │      runReviewGateStep(...)   ← pre-exec gate
                           │      runExecuteStep(...)      ← browser flow
                           │      runVerifyStep(...)       ← verify again
                           │
                           ├─▶ NEW post-exec review.requested lands
                           │
                           └─▶ await next decision (4-decision model — Week-2a):
                                 ├─ approve   → run.completed (status=ok)
                                 ├─ edit      → treated as approve (UI hides
                                 │              Edit on post-exec to prevent
                                 │              a wedge; server semantic is
                                 │              edit ≡ approve on this gate)
                                 ├─ reject    → backtrack loop (re-run
                                 │              block1 → reviewGate →
                                 │              execute → verify with
                                 │              carried observations)
                                 │              (cap MAX_BACKTRACKS=2)
                                 ├─ terminate → return {success: false,
                                 │              skipped: true}; logAndNotify
                                 │              emits status=rejected via
                                 │              the skipped-derivation path
                                 └─ cap trip  → {success: false,
                                              skipped: true} — same
                                              skipped-derivation path as
                                              terminate (Finding 2 fix
                                              from Commit A)

         Pre-exec Terminate skip-guard (Week-2a gate-decision-model-hotfix-1):
           humanVerifyGateStep entry checks `if (inputData.skipped) return
           inputData;` before any bus work. When pre-exec Terminate cascades
           through executeStep.skipped → verifyStep.skipped, the post-exec
           gate does NOT open — skipped VerifySchema flows through verbatim
           and logAndNotify's skipped-derivation emits status=rejected.


Both loops are structurally symmetric. They use the SAME envelope variant
(block.backtrack.triggered) discriminated by `fromStep`. They BOTH pass
observations via runBlock1's `seedObservations` opt, NEVER via a
withRunContext spread.

 9. HOW TO EXTEND — COMMON RECIPES
================================================================================

9.1  ADD A NEW TOOL (e.g., Slack client, ticket-system API, a new RAG index)
────────────────────────────────────────────────────────────────────────────

  1. Create `src/mastra/tools/<tool>.ts`.
  2. Implement the tool's functions. Each one should:
       a. accept `{ runId, bus, stepId }` in its opts (plus whatever else);
       b. emit `tool.started` with an invocationId + name + args;
       c. do the actual work (HTTP call / subprocess call / etc.);
       d. emit `tool.completed` (with resultSummary + durationMs) OR
          `tool.failed` (with error);
       e. throw a typed error class on failure.
  3. If the external service is unreliable, wrap it in a circuit breaker:
         const breaker = getCircuit("<service-name>");
         await breaker.execute(() => httpCall(...));
  4. If you want to use this tool inside a ReAct step, wrap it as a
     `ReactTool` (see `src/mastra/lib/reactRunner.ts` for the shape)
     and include it in the tools list passed to `createReActStep`.


9.2  ADD A NEW ENVELOPE FRAME VARIANT
──────────────────────────────────────

  1. In `src/events/envelope.ts`:
       a. Define the Zod schema for your frame payload (extend
          `timelineHeader` if it's a regular frame);
       b. Add it to the `TimelineFrameSchemas` array (or
          `TransportFrameSchemas` for seq:null frames).
  2. In the reviewer UI (separate service, `services/test-webapp`):
       a. If it should render, extend the projection function in
          `page.tsx` to handle the new `type`;
       b. Add render component + CSS.
  3. Any emitter just calls `bus.publish({ ... type: "your.type", ... })`.
  4. NO envelope-test additions needed for simple additive frames — the
     size-guard + discriminated-union tests catch schema drift.


9.3  ADD A NEW WORKFLOW STEP
─────────────────────────────

  1. In `src/mastra/workflows/triage.ts`:
       a. Define the input/output Zod schemas if new;
       b. Implement an extracted body function `runMyStep(inputData, ...)`
          — pattern: call `getRunContext()` for {runId, bus}, emit frames
          around interesting work, return typed output;
       c. Wrap with `createStep({ id, inputSchema, outputSchema,
          execute: ({ inputData, abortSignal }) => runMyStep(...) })`;
       d. Add the step to the `triageWorkflow` chain via `.then(myStep)`;
       e. Update `StepIdSchema` in envelope.ts with the new stepId.
  2. The reviewer UI's LEFT column reads `step.completed.output` — if
     you want a custom body renderer, add a case to `renderXxxBody`.


9.4  ADD A NEW HUMAN GATE
──────────────────────────

  1. Envelope: add your new stepId to `GateStepIdSchema` (the narrow
     enum used by `clientFrameSchema.review.decide.stepId`).
  2. Envelope: add the corresponding stepId to `StepIdSchema` too.
  3. Decide 4-decision dispatch semantics for your gate. The Week-2a
     gate-decision-model gives you four client-side inputs; you MUST
     map each to a workflow action:
         approve   — typically "proceed to next step in the chain"
         reject    — typically "replan without reviewer notes" (refine
                     with an auto-generated directive observation); on
                     post-exec-style gates this is the backtrack entry
                     point
         edit      — typically "replan with reviewer's patch.notes"
                     (refine with notes as seed observations); HIDE on
                     gates where edit ≡ approve to prevent wedge states
                     (see humanVerifyGateStep for the pattern)
         terminate — ALWAYS a skip-cascade to logAndNotify with
                     status=rejected; never loops, never replans
  4. In `triage.ts`, implement the step body. Example pattern matches
     runReviewGateStep / humanVerifyGateStep:
         bus.publish({ runId, stepId, payload:
           { type: "review.requested", plan, screenshots, viewerUrl,
             requiresApproval: true, reviewHint: "my_phase" } });
         const decision = await bus.awaitDecisionForStep(runId, "<stepId>");
         if (decision.decision === "approve") { /* proceed */ }
         if (decision.decision === "terminate") { /* skip-cascade */ }
         if (decision.decision === "reject")   { /* replan w/o notes */ }
         if (decision.decision === "edit")     { /* replan w/ notes */ }
     IMPORTANT: if your gate's implementation path cascades skipped=true
     upstream (like pre-exec Terminate does through executeStep /
     verifyStep), place a skip-guard at the gate's entry:
         if (inputData.skipped) return inputData;
     BEFORE any bus.publish, decision await, or local state init. See
     humanVerifyGateStep's Week-2a hotfix-1 guard at triage.ts:1976-1978
     for the canonical shape.
  5. Register the new stepId in the workflow chain via `.then(myGate)`.
  6. In the reviewer UI (services/test-webapp), extend the ChatBar
     state machine if your gate has distinct copy. The ChatBar reads
     `pendingReview.reviewHint` to branch Submit-button labels and
     body copy; add a new reviewHint enum value if your gate is
     semantically distinct from pre_exec / post_exec (envelope.ts
     ReviewRequestedFrame.reviewHint).
     - If edit ≡ approve on your gate: hide the Edit button via
       `isPostExec` or equivalent guard (copy the pattern from
       humanVerifyGateStep's wedge-prevention docblock).
     - Terminate link is always available via `canTerminate` and
       needs no per-gate wiring — the two-step confirm UI is generic.


9.5  APPLY REACT TO AN EXISTING STEP
─────────────────────────────────────

  E.g., you want `classify` to do RAG lookups during classification.
  Today it's a one-shot Haiku call.

  1. In `triage.ts`, replace `runClassifyStep`'s body with a

9.7  ADD A NEW MCP SERVER (filesystem, shell, Slack, GitHub, DB, etc.)
──────────────────────────────────────────────────────────────────────

  §9.1 covers generic tools (in-process calls, HTTP clients). An MCP
  server is a SUBPROCESS you spawn and talk to over stdio via
  `@modelcontextprotocol/sdk`. The shape is different enough to warrant
  its own recipe. The CANONICAL reference implementation is
  `src/mastra/tools/playwrightMcp.ts` — copy its structure.

  NOTE: use raw `@modelcontextprotocol/sdk` only. Do NOT introduce
  `@mastra/mcp`; the agent's MCP integration is deterministic-callTool
  semantics by design (see the briefing's §4.6 invariant).

  Steps:

  1. Create `src/mastra/tools/<name>Mcp.ts`. Copy the skeleton from
     `playwrightMcp.ts`:
       - `launchSession({ runId, bus, stepId, signal })` that spawns the
         MCP server subprocess, creates an `StdioClientTransport`, and
         returns a session object with typed `callTool()` methods.
       - Four INJECTABLE hooks for unit testing without spawning real
         subprocesses: `clientFactory`, `transportFactory`,
         `killProcessTreeImpl`, `fsImpl`. Copy the shapes verbatim.
       - `session.close()` tree-kills the subprocess via a BFS + SIGKILL
         leaf-first walk (the `killProcessTree` helper in
         playwrightMcp.ts). MCP servers routinely reparent helpers to
         PPID=1 on SIGTERM; leaking them is the #1 foot-gun.

  2. Frame emission — identical contract to §9.1:
       - Wrap every `callTool()` in `tool.started` + `tool.completed` /
         `tool.failed` with `invocationId` + `name` + `durationMs`.
       - REUSE existing frame types where the semantic fits (`tool.*`,
         `browser.*`, `rag.retrieved`). Do NOT invent new frame types
         unless the MCP surfaces a genuinely new category — the reviewer
         UI reads frame types and adding one is scope creep.
       - If `callTool()` returns a large payload (diff, dataset page,
         logs), persist to disk and emit a frame with a file path
         reference. The `browser.screenshot` pattern at
         `/workspace/.playwright-videos/<runId>/<seq>.png` is the
         template. Keeps every frame under the 16 KiB envelope cap.

  3. Session lifetime decision — two choices, each with invariant
     implications:

     (a) PER-CALL session (spawn → callTool → close inside one step):
         Simplest. No RunContext changes. No CTX SPREAD INVARIANT
         concerns. Use this when the MCP is fast to spawn (<500ms) and
         doesn't need to carry state (auth, workspace root) between
         steps.

     (b) LONG-LIVED session across workflow steps (like `ctx.browser`):
         Add a new field to `RunContext` in `src/mastra/runContext.ts`.
         This field IS load-bearing under the CTX SPREAD INVARIANT:
         any step that MUTATES `ctx.<yourField> = session` must run
         OUTSIDE any `withRunContext({ ...ctx, ... })` spread. Mirror
         `ctx.browser`'s treatment: dry_run mutates it in the OUTER
         scope (see Block 1 Design Invariant #2 in §4), and every
         downstream consumer reads from the outer scope. If your MCP
         session needs a pre-close before re-launch (see Bug B in
         7b.iii.b series), implement it at the mutation site, not the
         read site.

         SESSION-REUSE PATTERN (Week-2b-runtime — canonical reference:
         `runExecuteStep` at `src/mastra/workflows/triage.ts`):

         When a session-bearing tool (like Playwright MCP, but the
         pattern applies to any stateful MCP — filesystem working-dir,
         LSP client, authenticated API client, etc.) is populated by
         an UPSTREAM step (dry_run) and consumed by a DOWNSTREAM step
         (execute) with a HUMAN GATE between them, the downstream
         consumer should:

           1. Liveness-probe the existing session BEFORE launching a
              fresh one. Cheap probe = the cheapest read-only call the
              wrapper exposes (e.g. `session.snapshot()` for Playwright,
              a `fs.stat` for a filesystem MCP, a `noop/ping` for an
              API MCP). ~100-300ms cost; wrap in try/catch.

           2. Tag the session with the CURRENT step's `stepId` BEFORE
              probing so the probe's `tool.started` / `tool.completed`
              frames attribute correctly (not to the stale upstream
              stepId). On probe failure the retagged session is about
              to be closed anyway; the mutation costs nothing.

           3. Branch on the probe outcome:
                - SUCCESS → reuse the session. If the downstream step
                  has skill-card-driven dispatch, pass a "resume at
                  first `destructive: true` step" flag so the executor
                  skips the non-destructive prefix the upstream step
                  already walked. For non-skill-card dispatchers, the
                  equivalent is "start from the first post-gate-boundary
                  action." Log at INFO level.
                - FAILURE → graceful degrade: close the dead session
                  (silent catch — profile lock releases either way),
                  `launchBrowser`-equivalent a fresh one, set
                  `resumeAtFirstDestructive: false`. Log at WARN level
                  (ops signal for tuning gate-wait vs session-TTL).

           4. The session-none branch (upstream step threw before
              launch) takes the fresh-launch path with no probe.

         Why the pattern matters:
           - Preserves UX: the happy path skips 3-5s of re-auth per
             run (a large fraction of runs in a fast-reviewer
             workflow).
           - Read-only skills (no destructive step) become a correct
             no-op at the consumer step — the skip-prefix loop never
             finds a destructive boundary, stepsRun=0, no wasted work.
           - Graceful under long reviewer waits: session TTL expiry
             does not crash the run; the fallback takes the same
             launchBrowser path unconditional fresh-launch would.
           - Observable via a single log signature per branch — ops
             can grep for `reusing live ... session` vs `... session
             is dead` to trend reviewer-wait-time against session TTL.

         What you do NOT do:
           - DO NOT unconditionally pre-close and re-launch at every
             gate transition. That throws away the session-reuse
             optimization that the gate-aware workflow was designed
             around. (Rejected as "Path A simple" during the Week-2b
             runtime review — docs/MASTER_PLAN.md captures the
             decision.)
           - DO NOT add a test-only DI seam to production code to
             cover the fallback branch in unit scope. The fallback
             is 8 LoC of straight-line logic that mirrors the
             upstream step's pre-close pattern; live smoke + polish-
             queue integration test is sufficient coverage.

  4. Consumption-pattern decision — two choices:

     (a) LLM-DRIVEN (the ReAct style): expose the MCP's tools as
         `ReactTool` entries (see `reactRunner.ts` for the shape) and
         include them in `createReActStep({ tools: [...] })`. The LLM
         decides when to call each tool. Good for cognitive steps
         (classify, verify) where the sequence is not known in advance.
         No YAML authoring required.

     (b) SKILL-CARD-DRIVEN (the deterministic style): extend the
         skill-card schema at `src/schemas/skill-card.ts` with a new
         `action:` type (or a new `tool:` dispatcher enum value). Teach
         `skillCardExecutor.ts` to dispatch that action/tool type to
         your MCP session's corresponding `callTool()`. The YAML
         recipes then compose your MCP's tools deterministically, same
         as Playwright recipes do today. Good for resolution flows with
         known, auditable step sequences.

  5. Trust boundary — domain-specific and REQUIRED. Playwright has
     single-origin pinning; your MCP needs its equivalent:
       - Filesystem MCP:   root-allowlist (never escape a sandbox dir).
       - Shell MCP:        command-allowlist (or denylist — pick one
                           consciously; prefer allowlist).
       - API MCP:          scoped token, narrow OAuth scopes, per-run
                           rate limit.
     Declare the boundary in the wrapper module's docblock. Credentials
     live in env vars (Zod-validated in `src/env.ts`); never pass
     plaintext creds through LLM prompts.

  6. Circuit breaker — optional but cheap. Wrap per-callTool
     invocations in `getCircuit("<service-name>").execute(...)` if the
     MCP fronts a flaky external (network APIs, rate-limited services).
     Subprocess spawn itself usually doesn't need it.

  7. Tests — `playwrightMcp.ts`'s test file is the template. With the
     four injectable hooks you can unit-test spawn args, teardown order,
     abort propagation, and frame shape WITHOUT actually launching the
     MCP subprocess. Aim for parity with its coverage (12 tests today).

  What you do NOT touch when adding an MCP: the orchestration engine,
  workflow chain, reviewer UI, gates, meta-loops, primitives layer.
  Adding an MCP is a pure tool-layer addition — per the substrate map
  in `docs/PLATFORM_PIVOTS.md §3`, the rest of the stack doesn't care
  what's on the other end of `callTool()`.

11. INVARIANTS TO REMEMBER (the things that bite if broken)
================================================================================

  1. FRAMES ARE THE SOURCE OF TRUTH. If the UI needs to know something,
     emit a frame for it. Do not smuggle data through side channels.

  2. CTX SPREAD INVARIANT. `withRunContext({ ...ctx, <override> }, fn)`
     creates a NEW object. Mutations in `fn` are LOST. Specifically:
     NEVER wrap `runBlock1` in such a spread. Thread observations via
     `runBlock1(..., { seedObservations })` instead. (Full rules in the
     docblock at the top of `runContext.ts`.)

  3. STEPID NAMESPACE. Extracted step bodies invoked OUTSIDE Mastra's
     engine (Block 1 controller, refine loop, backtrack loop) must emit
     their own `step.started` / `step.completed` frames manually. The
     stepEmitter only fires for Mastra-invoked steps.

  4. 16 KIB FRAME CAP. No frame may exceed MAX_FRAME_BYTES. Large payloads
     (screenshots, full plan text) must be persisted to disk/DB and
     referenced by path in the frame. Bus synthesizes a `run.failed` if
     violated.

  5. BUS DECISION SLOTS ARE PER-STEPID. `awaitDecisionForStep(runId,
     stepId)` — NOT the deprecated single-slot `awaitDecision(runId)`.
     This is what enables the two-gate architecture.

  6. DB CROSS-CHECK IS AUTHORITATIVE, NOT THE WIRE. A run that reports
     `status=ok` on the WS stream might still have stepsRun=0 + Jane's
     row unchanged. Smoke acceptance requires querying Postgres or the
     actual target system. (See STARTUP_PROCESS §1.6.7.)

THE REFRAMING
-------------
Pre-week2d: skill cards at `kb/skill_cards/<app>/<skill>.yml` were
RUNTIME WALKERS. `runDryRunStep` + `runExecuteStep` both called
`executeSkillCardSteps(loadSkill(plan.skillCardIds[0]))` to walk the
YAML-declared action sequence against the live browser.

Post-week2d: skill cards are SCAFFOLDS. Agent's actual trace is the
ARTIFACT. Divergence between them is the auditability signal.

  ┌──────────────────────────────┬──────────────────────────────┐
  │ SCAFFOLD                     │ ARTIFACT                     │
  │ (pre-authored, versioned)    │ (ephemeral, per-run)         │
  ├──────────────────────────────┼──────────────────────────────┤
  │ kb/skill_cards/<app>/*.yml   │ ctx.tempSkillCard (in-memory)│
  │                              │ + materialized_skills row    │
  │                              │ + step.completed frame       │
  │                              │                              │
  │ Authored by humans           │ Written once by              │
  │                              │ runMaterializeSkillCardStep  │
  │                              │                              │
  │ Read by plan (selection)     │ Read by runExecuteStep       │
  │                              │      runVerifyStep           │
  │                              │                              │
  │ Authoritative for:           │ Authoritative for:           │
  │  - intent shape              │  - what was ACTUALLY done    │
  │  - destructive boundaries    │  - deterministic execute walk│
  │  - declared inputs           │  - postcondition evaluation  │
  │  - postconditions            │    basis                     │
  │                              │                              │
  │ Example:                     │ Example:                     │
  │  reset_password.yml →        │  reset_password_materialized │
  │   steps[] + destructive:true │   steps: [... actionTrace ...│
  │   on "Reset password button" │   + appended click on        │
  │                              │     boundaryReached.element] │
  └──────────────────────────────┴──────────────────────────────┘

THE FLOW
--------
   1. plan selects scaffold (by skillCardIds[]) + emits plan.inputs
        + optionally plan.targetUrl (week2e)
   2. dry_run: agentic ReAct exploration using
        `buildBrowserReactTools()`. Agent sees scaffold as HINT (via
        formatScaffoldHint) but is free to adapt to today's UI.
        Stops at `boundary_reached` (see §5 REACT_FINAL_SENTINEL).
   3. review_gate: reviewer sees actionTrace + boundaryReached +
        divergence banner (if scaffoldMatch=false).
   4. materialize (on approve): scaffold-inherited postconditions +
        actionTrace (with verbatim-input substitution) + appended
        destructive click (from boundaryReached.element) →
        ephemeral Skill → ctx.tempSkillCard + Postgres row with
        name `<host>_<scaffold>_<uuid4>`.
   5. execute: `executeSkillCardSteps(ctx.tempSkillCard, ...)` with
        resumeAtFirstDestructive:true (session-reuse path from
        dry_run; destructive-append is the appended last step).
   6. verify: structured LLM-judge on tempSkillCard.postconditions
        vs fresh snapshot → {success, evidence[]} with specific
        DOM-ref citations.

CRITICAL MENTAL MODEL — THE DESTRUCTIVE-APPEND
----------------------------------------------
`boundary_reached` fires BEFORE the destructive click. Therefore
`DryRunSchema.actionTrace` contains ONLY non-destructive pre-
destructive steps. The destructive action exists ONLY as metadata
in `boundaryReached.element` / `.reason`. The materializer APPENDS
a synthesized click step marked `destructive:true` as the final step
of the ephemeral skill. A naive 1-to-1 mapping would produce a zero-
destructive-step skill — `executeSkillCardSteps(resumeAtFirstDestructive:
true)` would no-op on it.

See `materializeSteps()` in `triage.ts` for the implementation.
Regression guard: `materializeSkillCard.test.ts [1]`.

DIVERGENCE
----------
When `boundaryReached.scaffoldMatch === false`, the materializer
populates `MaterializeSchema.divergence: { expected, actual, reason }`:
- `expected`: scaffold's declared destructive element name
- `actual`: agent's observed element name
- `reason`: agent's explanation

Reviewer UI (Part 4) renders:
- LEFT column: warn-yellow chip on the materialize_skill_card row
- RIGHT feed: full-width warn banner with expected/actual/reason

Divergence is signal, not error — it's legitimate UI drift and the
reviewer decides whether the agent's observed action is semantically
equivalent to the scaffold's declared one.

WHAT STAYS FROM WEEK2B
----------------------
- Skill-card schema (`src/schemas/skill-card.ts`) — unchanged; temp
  cards conform to the same Zod schema.
- `loadSkill` + `loadAllSkills` — used by plan for scaffold selection.
- `executeSkillCardSteps(... preflight: false)` — used by execute to
  walk the temp card.
- `kb/skill_cards/test-webapp/*.yml` — become scaffolds; still
  validated by `validate:skill-cards`, still embedded to RAG by

WHAT CHANGED IN WEEK2D
----------------------
- `executeSkillCardSteps(... preflight: true)` in `runDryRunStep` —
  removed. Replaced by ReAct loop over `buildBrowserReactTools()`.
- `runDryRunStep` body — rewritten as `runReActIterations(...)`.

13. CONCEPTUAL PRIMITIVE — REVIEWER-CONTROL VIA EDIT-REFINE (week2d/2e)
================================================================================

The product's pitch: "reviewer steers the agent via Edit at the gate."
This primitive codifies the mechanism so every layer treats it
consistently.

THE PATTERN
-----------
  reviewer's Edit note (patch.notes)
     │
     ▼
  runReviewGateStep refine loop wraps note as seedObservation:
     "Pre-exec refine N: reviewer note: <note>"
     │
     ▼
  runBlock1(input, deps, { seedObservations: [...] })
     │  controller seeds its `observations` accumulator on entry
     ▼
  inner cognitive spread (classify/retrieve/plan) reads
     priorObservations via ambient ctx — observations flow into
     each LLM call's user message via `observationsPrefix(obs)`
     │
     ▼
  runDryRunStep reads priorObservations via its deps param
     (Part 2 hotfix-1 — threaded explicitly because dry_run runs
     OUTSIDE the cognitive spread per CTX SPREAD INVARIANT)
     │
     ▼
  AGENT ADAPTS: Sonnet in classify/retrieve/plan reshapes
     classification, retrieval queries, plan.inputs, plan.targetUrl,
     plan.skillCardIds based on reviewer guidance. ReAct dry_run
     agent reshapes browser exploration (different fillForm values,
     different search terms, different navigation).

AUDITABILITY REQUIREMENT — Path A+
----------------------------------
Implicit behavior shifts (dry_run just navigates somewhere different)
are NOT sufficient for the audit story. Corrections must produce
DISCRETE, VISIBLE FIELD DELTAS in the event log.

Pattern: plan's system prompt INSTRUCTS Sonnet to re-emit corrected
values as first-class plan output fields when observations contain
corrections. Examples:

  Correction         → plan field that must flip       → Reference
  ─────────────────    ─────────────────────────────    ──────────────
  Target user         plan.inputs.email                 week2d Part 3
  Operator creds      (agent-level via ReAct)           week2d hotfix-2
  Element name        (boundaryReached.element)         week2d Part 4
                      + reviewer approves divergence
  Target URL          plan.targetUrl                    week2e Path A+
  Skill selection     plan.skillCardIds[0]              future
  (future)

The plan prompt contains explicit blocks ("TARGET URL RESOLUTION",
"INPUTS EXTRACTION") telling Sonnet "if prior observations contain
a correction of type X, adopt it authoritatively and emit as field
Y." Without these blocks, Sonnet might adopt the correction
implicitly (via its own tool choices) but the audit log wouldn't
show a discrete before/after.

LIVE-VERIFIED AXES
------------------
These three reviewer-correction types have been live-smoked end-to-end:

  1. Target user flip (jane → leo) via inputs
     - Smoke runId: `153b7084-f7bc-4c8f-a477-596af6a985df`
     - Mechanism: seedObservations → plan.inputs re-extracted

  2. Operator creds flip (theo → sam) via dry_run adoption
     - Same smoke (combined with user flip)
     - Mechanism: seedObservations → Sonnet's dry_run ReAct
       reshapes fillForm values in browser exploration

  3. Target URL flip (planetfitness.com → test-webapp:3000)
     - Smoke runId: `6f630744-4785-46b9-b1e5-b689b49bcfaa`
     - Cross-ORIGIN flip, not just path correction
     - Mechanism: seedObservations → Sonnet re-emits
       plan.targetUrl per Path A+ prompt instruction
     - Evolution in event log:
         pass 0: https://planetfitness.com (ticket)
         pass 1: https://planetfitness.com (Block 1 retry)
         pass 2: https://planetfitness.com (Block 1 retry)
         refined: http://test-webapp:3000 (post-Edit)

BUDGETS
-------
Refine loop is capped at `MAX_PRE_GATE_REFINES = 2` (3 total review
cycles per gate). Post-exec reject triggers `MAX_BACKTRACKS = 2`
backtrack cycles. Budget-exhaustion flows through the skip-cascade
to `status: rejected` (NOT `status: failed`) per week2a Finding 2.

WHEN NOT TO USE EDIT
--------------------
- Block 1 exhausted (`blockResult.passedLast === false`): Edit hides
  in the UI because same upstream cause retriggers the refine budget.
  Reviewer picks Reject (replan with directive) or Terminate.
- Post-exec gate: Edit hidden in the UI; `humanVerifyGateStep` treats
  edit ≡ approve server-side. "Approve + note" semantic is polish #11.
- Hopelessly lost: Terminate is the escape. MAX budgets cap refine
  iteration cost; reviewer shouldn't indefinitely retry.
